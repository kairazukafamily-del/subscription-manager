# Retrospective.md — subscription-manager

作業日: 2026-05-08

---

## 成果物の出来ばえ

### 良かった点

- 機能面は仕様通り全て動作している（CRUD・為替・通知・フィルタ）
- デザインの一貫性は高い。my-hp の色・余白・タイポグラフィを正確に継承できた
- サーバーサイドプロキシ＋CDNフォールバックの構成は、個人ツールとしては過剰にならない範囲で堅牢

### 反省点（実装側）

- **↻ボタンの強制再取得が実装されていなかった** — Spec に明記されていたにもかかわらず、実装時に見落とした。セルフチェックが不十分だった
- **カード換算表示のスペース（`$N × ¥N` vs `$N × ¥ N`）** — Spec と実装が微妙にずれたまま最後まで気づかなかった
- **カテゴリ機能を丸ごと削除した誤解** — 今回最大のミス。「カテゴリって部分」という言葉を機能と解釈してしまった。作業量・影響範囲が大きい変更をする前に「UIのテキスト削除ですか、機能ごと削除ですか？」と一度確認すべきだった

---

## 指示・コミュニケーションについて

### 明確で良かった点

- 「frankfurter.appで行きます」「月額のみ」「個人用だから〜は不要」など、理由付きで決断が速かった。実装が止まらなかった
- 「英語じゃなくて日本語統一」など方針転換も明確で、揺れが少なかった
- 「SPECと実装の乖離チェック」など、品質確認の観点が自発的にあった

### 改善できたかもしれない点

- **「カテゴリって部分を削ぎ落として」** — この一言が唯一の大きなロールバックを生んだ。「〜という文字を消して」「〜という機能を消して」と対象を明示すると防げた
- **UIの調整指示が複数まとめて来ることがあった** — 例：「subscriptionsはセンスいい / もう少し大きく / すべて・アクティブ・停止中の余白 / 解約済み不要 / カテゴリ編集 / 英語→日本語」を一度に。優先順位が不明なまま実装することになった。一度に3つ以上の変更は、どれかが意図とずれるリスクが上がる

---

## 総評

指示の質は全体的に高く、決断も早かった。ロールバックが1回あったが、それ以外はスムーズに進んだ。実装側のミスは「確認コストより実装コストを優先して動いてしまった」場面が原因で、指示が曖昧だったというより、「解釈して進む」より「確認して進む」を選ぶべき場面を見誤った点が反省。
